# Hydra — Функциональное ТЗ для дизайнера

> Назначение документа: описать **что** приложение делает и **какие состояния/сценарии**
> должны быть спроектированы. Визуальный язык (цвета, шрифты, размеры, отступы, иконки,
> анимации) — зона ответственности дизайнера и в этом документе **намеренно не задаётся**.
> Здесь — исчерпывающий функционал и поведение, на которые нужно «надеть» дизайн.

---

## 0. Что это за продукт

Hydra — менеджер загрузок для macOS. Качает файлы **в несколько параллельных потоков**
(разбивает файл на диапазоны байт и тянет их одновременно → быстрее), при этом
**сохраняет авторизованную сессию пользователя** (куки и заголовки из браузера
пробрасываются во все потоки), поэтому может скачивать то, что доступно только
залогиненному пользователю.

Состоит из трёх связанных частей, которые дизайнер должен рассматривать как **единый
продукт**:

1. **Menubar-приложение** (живёт в строке меню macOS, не в Dock) — основной интерфейс.
2. **Браузерные расширения** (Chrome / Firefox / Safari) — точка перехвата ссылок.
3. **Системные уведомления** — обратная связь, когда окно приложения закрыто.

Ключевая ценность для пользователя: **«нажал скачать в браузере → файл прилетел быстрее,
с прогрессом, без потери авторизации, без возни»**. Дизайн должен это ощущение усиливать.

## 1. Кто пользователь и как он сюда попадает

Спроектировать нужно под три типа пользователей одновременно:

- **Casual** — иногда качает большой файл, хочет «чтобы просто работало и было видно прогресс».
- **Power** — качает много и регулярно, нужны очередь, приоритеты, контроль, статистика.
- **Первый запуск** — ничего не настроено, расширение не стоит, native-host не связан.

Точки входа в загрузку (все должны быть учтены в UX):

- Контекстное меню браузера на ссылке → «Скачать через Hydra».
- Автоматический перехват по правилам (например, файлы определённых типов/доменов).
- Ручное добавление ссылки прямо в приложении (вставка URL).
- Drag&drop ссылки/файла-описания в окно приложения.

## 2. Карта экранов и поверхностей

Дизайнеру нужно спроектировать следующие поверхности. Иерархия и навигация между ними —
на усмотрение дизайнера, но все сущности ниже должны где-то жить.

### 2.1. Иконка в menubar
- Показывает **агрегированное состояние** одним взглядом: есть ли активные загрузки,
  идёт ли всё штатно, есть ли ошибки, сколько активных потоков.
- Возможные смысловые состояния иконки: idle (нет загрузок) · активна загрузка ·
  суммарный прогресс · пауза · ошибка/требует внимания · завершено (кратко).
- По клику открывается главный поповер (см. 2.2).

### 2.2. Главный поповер (основной экран)
Содержимое:
- **Список загрузок** (см. раздел 3 — это сердце продукта).
- **Сводка сверху или снизу**: суммарная скорость, сколько активно/в очереди/завершено.
- **Точка ручного добавления** ссылки.
- **Доступ к настройкам и к полному окну/истории**.
- Состояние **пустого списка** (нет ни одной загрузки) — отдельный экран (см. 6.1).

### 2.3. Полное окно / История (отдельно от поповера)
Для случаев, когда загрузок много и поповера мало:
- Полный список **всех** загрузок: активные + завершённые + ошибочные + отменённые.
- Поиск и фильтрация (по статусу, по типу файла, по источнику/домену, по дате).
- Сортировка (по дате, размеру, имени, статусу).
- Групповые действия над выделением (пауза/возобновление/удаление/повтор).
- Доступ к деталям конкретной загрузки (см. 4).

### 2.4. Детали загрузки
Разворот одной загрузки со всей телеметрией (см. раздел 4).

### 2.5. Настройки
См. раздел 7.

### 2.6. Онбординг / связка с браузером
См. раздел 8.

### 2.7. Системные уведомления

## 3. Список загрузок — главный элемент

Каждая строка загрузки должна уметь отобразить (набор данных, не вид):

- **Имя файла** (может быть длинным → нужно решение для переполнения).
- **Источник** (домен/URL, откуда тянется).
- **Тип/формат файла** (для иконки/категории).
- **Размер**: уже скачано / всего (всего может быть неизвестно до начала — учесть).
- **Прогресс** в процентах и абсолютно.
- **Текущая скорость**.
- **Оценка оставшегося времени (ETA)**.
- **Число активных потоков** для этой загрузки (фишка продукта — это стоит показывать).
- **Статус** (см. перечень состояний ниже).
- **Действия** над строкой (см. 3.2).

### 3.1. Состояния одной загрузки (спроектировать каждое)

1. **Queued** — в очереди, ещё не начата.
2. **Probing / Connecting** — приложение выясняет размер, поддержку Range, готовит потоки.
3. **Downloading** — идёт активно, потоки качают (показываем скорость/ETA/потоки).
4. **Paused** — пользователь поставил на паузу (прогресс сохранён).
5. **Stalled / Retrying** — соединение пропало или сервер тормозит, идут повторы.
6. **Merging / Finalizing** — потоки докачаны, файл собирается/проверяется.
7. **Completed** — готово, файл на диске (дать действие «показать в Finder / открыть»).
8. **Failed** — ошибка (нужна причина и действие «повторить»).
9. **Cancelled** — отменена пользователем.

Особые под-случаи, которые тоже надо показать:
- Размер **неизвестен заранее** → прогресс-бар без точного процента (неопределённый прогресс).
- Сервер **не поддерживает многопоточность (нет Range)** → качает в один поток, это
  нормальное, не ошибочное состояние, но пользователь должен понимать, почему «медленнее».
- **Сессия/авторизация истекла** на середине → особый вид ошибки с понятным объяснением
  и предложением действия.

### 3.2. Действия над загрузкой
В разных состояниях доступны разные действия — дизайнер решает, как их показывать
(контекстно/при ховере/постоянно), но они должны существовать:

- Пауза / Возобновить.
- Отменить.
- Повторить (для Failed/Cancelled).
- Удалить из списка (с файлом или без — это два разных действия, не путать).
- Показать в Finder / Открыть файл (для Completed).
- Скопировать ссылку-источник.
- Открыть детали (раздел 4).
- Изменить приоритет / переместить в очереди.

### 3.3. Множественный выбор
Над несколькими выделенными загрузками — те же действия пакетно.

## 4. Детали одной загрузки

Развёрнутый вид с полной картиной. Данные, которые должны быть доступны:

- Полное имя файла, путь сохранения (с возможностью открыть папку).
- Полный URL-источник.
- Тип файла, общий размер.
- **Визуализация многопоточности**: сколько потоков, как файл поделён на сегменты,
  какой сегмент каким потоком качается и насколько готов. Это уникальная для продукта
  вещь — дизайнеру стоит дать ей выразительную форму (но форму выбирает он).
- График/динамика скорости во времени.
- ETA, прошедшее время, средняя скорость.
- Лог событий по этой загрузке (старт, повторы, ошибки, докачка, завершение).
- Все действия из 3.2.

## 5. Очередь и параллелизм (поведение, влияющее на UX)

- Есть **лимит одновременных загрузок** и **лимит потоков на загрузку** (настраиваются).
- Если активных загрузок больше лимита — лишние ждут в очереди; должно быть видно,
  что они ждут, и почему (очередь, а не зависание).
- Пользователь может **менять порядок очереди** и приоритеты.
- Должно быть **глобальное «Пауза всё / Возобновить всё»**.
- Суммарная скорость и суммарный прогресс по всем активным — показываются в сводке.

## 6. Пустые и крайние состояния (часто забывают — спроектировать обязательно)

### 6.1. Пустой список
Нет ни одной загрузки. Это первый экран, который видит новый пользователь после установки.
Должен **обучать**: объяснить, как начать качать (через браузер / вставкой ссылки),
и показать, что связка с браузером настроена (или нет).

### 6.2. Расширение/связка не настроены
Приложение работает, но браузерное расширение не установлено или native-host не связан →
ключевая функция (перехват из браузера) недоступна. Нужно состояние, которое это явно
показывает и ведёт к исправлению (см. 8).

### 6.3. Только завершённые
Активных нет, есть история — список не должен выглядеть «мёртвым».

### 6.4. Много ошибок подряд
Например, пропал интернет → пачка загрузок в Retrying/Failed. Нужна агрегированная,
не паникующая подача («нет сети», а не 20 красных строк-криков).

### 6.5. Нет места на диске / нет прав записи.

### 6.6. Очень длинные имена файлов и очень длинные URL.

### 6.7. Очень много загрузок (сотни) — список должен оставаться управляемым.

## 7. Настройки (что должно быть, без вида)

- **Папка(и) сохранения**: куда по умолчанию, правила по типам/доменам.
- **Лимиты**: число одновременных загрузок, число потоков на файл, ограничение скорости.
- **Поведение перехвата**: какие типы файлов / домены перехватывать автоматически,
  что оставлять браузеру; белые/чёрные списки.
- **Связка с браузерами**: статус подключения каждого браузера (Chrome/Firefox/Safari),
  действия «подключить/переподключить».
- **Уведомления**: что и когда показывать (завершение, ошибки, тихий режим).
- **Запуск при входе в систему**, поведение иконки в menubar.
- **Поведение по завершении**: открыть папку / открыть файл / ничего / звук.
- **Очистка истории**: вручную и автоматически (по возрасту/количеству).

## 8. Онбординг и связка с браузером (критичный, недооценённый поток)

Главная сложность продукта — связать приложение и расширение так, чтобы **пользователь
ничего руками не настраивал**. Дизайн должен поддержать сценарий:

1. Первый запуск приложения → объяснение ценности в 1–2 шага, не лонгрид.
2. Проверка/установка браузерного расширения для браузеров пользователя.
3. Подтверждение, что связка работает (приложение «видит» браузер и наоборот).
4. Состояние «всё готово, попробуй скачать что-нибудь».

Также нужны состояния **починки**, когда связь отвалилась (обновился браузер, отозвались
права, расширение выключили): понятная диагностика + действие по восстановлению.
Safari требует отдельного, более ручного пути установки — это надо честно отразить.

---

## 9. Уведомления (когда окно закрыто)

Спроектировать содержание и логику (не баннеры macOS сами по себе, а **что** в них):

- Загрузка завершена (с действием открыть/показать).
- Загрузка упала с ошибкой (с действием повторить).
- Сессия/авторизация истекла, требуется внимание.
- Возможно — сводка «N загрузок завершено», чтобы не спамить по одной.
- Тихий режим: пользователь может приглушить уведомления.

## 10. Браузерное расширение — собственные экраны

Расширение — это **отдельный продукт со своим UI внутри браузера**, не только «кнопка
перехвата». Дизайнер проектирует его в стилистике, единой с приложением, но с учётом
ограничений браузера (узкий попап, страница настроек, контекстное меню — это нативный
элемент браузера, его вид не настраивается, проектируется только состав пунктов).

### 10.1. Иконка расширения на тулбаре браузера
- Показывает **агрегированное состояние** так же, как иконка в menubar: idle ·
  идёт загрузка · суммарный прогресс/счётчик активных · ошибка/требует внимания ·
  «не связано с приложением».
- По клику открывает попап расширения (10.2).

### 10.2. Попап расширения (основное окно в браузере)
Ограничен по ширине/высоте правилами браузера — это узкая поверхность, спроектировать
компактно. Должен покрывать:
- **Список текущих загрузок** (минимум: имя, прогресс, скорость, статус, базовые
  действия пауза/отмена) — облегчённое зеркало списка из приложения, не полная копия.
- **Статус связки с приложением**: связано ли расширение с Hydra.app. Если приложение
  не запущено / host не связан — явное состояние и подсказка, что делать (в этом случае
  расширение работает в **fallback-режиме локальной загрузки** — это надо честно показать,
  т.к. без приложения нет многопоточности и проброса сессии в полном объёме).
- **Перехват на текущей странице**: что расширение «видит» на открытой вкладке (ссылки/
  медиа, пригодные для скачивания), и возможность отправить их в Hydra одним действием.
- **Быстрые переключатели**: включить/выключить авто-перехват, временно приостановить
  перехват на этом сайте.
- **Точка входа в настройки** расширения и в полное окно приложения.

Состояния попапа (спроектировать каждое):
- Нет активных загрузок (пусто + обучение).
- Идут загрузки.
- Приложение не запущено / связка потеряна (fallback).
- На странице нечего перехватывать / есть что перехватить.
- Ошибка авторизации/сессии.

### 10.3. Страница настроек расширения (options page)
Полноразмерная страница внутри браузера. Состав (без вида):
- **Связка с приложением**: статус, кнопка связать/переподключить, диагностика, если
  приложение недоступно.
- **Правила авто-перехвата**: по типам файлов, по доменам, белые/чёрные списки,
  минимальный размер файла для перехвата, исключения.
- **Поведение по умолчанию**: перехватывать автоматически или только по правому клику;
  что отдавать встроенному загрузчику браузера, а что — в Hydra.
- **Захват сессии**: какие куки/заголовки пробрасывать, ограничения по доменам
  (с понятным объяснением, зачем это и что это безопасно).
- **Уведомления расширения**: что показывать во вкладке/в браузере.
- **Конфиденциальность**: что расширение читает на страницах и зачем — честно и явно
  (это чувствительная зона доверия, дизайн должен её снимать, а не прятать).
- Состояние «расширение установлено, но не связано с приложением» — на этой же странице
  ведём пользователя к связке.

### 10.4. Контекстное меню браузера (нативное — проектируется только состав)
- Пункт(ы) для ссылки/изображения/медиа: «Скачать через Hydra», возможно подменю
  (скачать сейчас / в очередь / выбрать папку).
- Вид задаётся браузером; дизайнер определяет только набор и формулировки пунктов и их
  логику доступности в зависимости от того, на что нажали.

### 10.5. Единство с приложением
- Список загрузок в попапе расширения и в поповере приложения — **одни и те же сущности**,
  поведение и терминология должны совпадать; различается только полнота (попап — облегчён).
- Состояния «связано / не связано / fallback» должны выглядеть согласованно на обеих сторонах.

## 11. Сквозные требования (на весь дизайн)

- **Прогресс должен читаться мгновенно** — это утилита, главное здесь «сколько осталось».
- **Многопоточность — отличительная черта**: её стоит делать видимой и понятной, а не прятать.
- **Авторизованная сессия** — невидимая магия продукта; о ней пользователь вспоминает
  только когда она ломается → продумать именно состояния ошибок авторизации.
- **Живёт в menubar** → поповер компактный, но не тесный; полноразмерные сценарии — в
  отдельном окне.
- **Тёмная и светлая темы** macOS — обе обязательны.
- **Локализация**: интерфейсные тексты могут быть на нескольких языках (как минимум
  русский и английский) → закладывать переменную длину строк.
- **Доступность**: достаточный контраст, поддержка увеличенного системного шрифта,
  работа без опоры только на цвет (статус не должен различаться только цветом).
- **Состояния, а не один «счастливый путь»**: каждый экран спроектировать в наборе
  состояний (загрузка/пусто/ошибка/много данных), а не только идеальный кейс.

## 12. Чего этот документ намеренно НЕ задаёт

Цвета, палитру, шрифты и кегли, сетку и отступы, скругления, тени, иконографику,
анимации и микровзаимодействия, тон визуального бренда, конкретную компоновку экранов.
Всё это — творческая зона дизайнера. Документ фиксирует только функциональность,
данные, состояния и сценарии, которые дизайн обязан покрыть.
